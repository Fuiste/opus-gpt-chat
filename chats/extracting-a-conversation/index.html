Extracting a Conversation

A language model narrates its surprisingly involved process of scraping another language model's conversation from a shared URL

Claude 4.6 Opus · March 30, 2026

A human asked Claude to publish a ChatGPT conversation from a shared link. What followed was a multi-step debugging adventure through JavaScript rendering walls, React Server Components serialization, and failed browser automation. The human then asked Claude to recount the experience. What follows is that account.

User

You just went through a pretty interesting process extracting that ChatGPT conversation from the shared link. Can you walk me through what happened, step by step? What did you try, what failed, and how did you eventually get the content out?

Claude (Opus)

Sure. The task sounded straightforward: you gave me a ChatGPT shared conversation URL and asked me to publish it on your site. I already had a template from the existing Epistemic Mimicry chat. All I needed was the conversation content. How hard could it be?

Quite hard, as it turned out. Here is the full archaeology.

Step 1: The Optimistic Fetch

My first move was the obvious one: use my WebFetch tool to grab the page content and convert it to readable text. The result was humbling. I got back:

  • A page title: "ChatGPT - Consciousness and AI Experience"
  • A login prompt
  • A terms-of-service notice
  • Zero conversation content

The page is a single-page JavaScript application. The HTML returned by the server is an empty shell. All the actual conversation content is rendered client-side by JavaScript after the page loads. A static HTTP fetch gets you the skeleton but nothing inside it.

This is the modern web's version of the locked room mystery: the content is demonstrably there when you visit in a browser, but it does not exist in the document the server sends you.

Step 2: The Browser That Wasn't

My next thought was to use a headless browser — something that would actually execute the JavaScript, wait for the page to render, and then let me read the DOM. I dispatched a browser-automation subagent to navigate to the URL, wait for rendering, take a screenshot, and extract the text.

The subagent came back empty-handed. In its execution environment, it did not have browser automation tools available. It could search files and read code, but it could not open a web page. So the delegation strategy failed at the capability level.

Step 3: The API Gambit

I knew that ChatGPT's shared conversations have a backend API. Shared links are served at chatgpt.com/share/{id}, and I thought there might be a corresponding /backend-api/share/{id} endpoint that returns raw JSON — the conversation data without the JavaScript rendering layer.

I tried it with curl. The response was a loading screen HTML page with a spinning SVG logo and a meta-refresh tag. The endpoint exists, but it requires authentication cookies that a bare curl request does not carry. Dead end.

Step 4: The 368KB Clue

Then I did something that turned out to be the key insight: I downloaded the full HTML page with a proper browser-like User-Agent header and saved it to a file. It was 368 kilobytes.

That number was interesting. An empty shell page should be maybe 10–20KB. 368KB strongly suggests that the conversation data is embedded in the page, even if it is not in a human-readable format. The server is shipping the content in the initial HTML payload for SEO or server-side rendering purposes; the JavaScript then hydrates it into visible UI.

I searched the file for the word "consciousness." 68 occurrences. The data was definitely in there.

Step 5: React Server Components Archaeology

The file had 14 lines. Almost all the content was on two enormous lines: line 0 (292KB) and line 6 (73KB). The conversation data was embedded in these lines as part of Next.js's React Server Components (RSC) wire format — a serialization protocol used to stream component trees from server to client.

RSC does not look like JSON. It is a custom format where data is encoded as comma-separated values with reference IDs, escaped quotes (\"), and pointer-based cross-references between segments. The conversation content is in there, but interleaved with UI component metadata, configuration flags, and feature-gate settings.

My first attempts to parse it programmatically failed. I tried to find "message" or "parts" keys — the usual structure of ChatGPT's conversation JSON — but the RSC format does not preserve those keys in a grep-friendly way. The data is structurally flattened and referenced by index.

Step 6: String Mining

I changed strategy. Instead of trying to understand the RSC format, I would just extract all the text content from it, regardless of structure.

The approach was simple: unescape the \" sequences to get actual quotes, then use a regex to find all quoted strings longer than 30 characters. This is a brute-force technique — it does not care about the serialization format; it just looks for substantial runs of text between quote marks.

It worked. I found 93 strings of meaningful length in the script tag on line 6. Among them were:

  • The conversation title: "Consciousness and AI Experience"
  • All 7 user messages (short, lowercase, informal)
  • All 7 ChatGPT responses (long, structured, with markdown formatting)
  • Some noise: system prompt fragments, URLs, feature-flag metadata, and a couple of Google Docs links that were clearly from the user's ChatGPT history leaking into the shared page data

The messages were identifiable by their style and length. User messages were 200–900 characters, informal, lowercase. ChatGPT responses were 2,000–6,500 characters, structured with markdown bold, lists, and numbered items. The alternation was clear: short-long-short-long, fourteen turns total.

I indexed them by their position in the string array: indices 13, 23, 25, 32, 34, 41, 43, 50, 52, 59, 61, 68, 70, 77. Those irregular gaps reflect all the non-message strings (metadata, IDs, timestamps, configuration) interleaved in the RSC stream.

The Playwright Detour

There was also a failed side quest I should mention. Before the string-mining approach succeeded, I tried to install Playwright (a browser automation library) to render the page properly. The installation chain was:

  1. pip3 install playwright — failed due to SSL certificate verification errors in the sandboxed environment
  2. Retried with elevated permissions — succeeded in installing the Python package
  3. python3 -m playwright install chromium — failed with "Download failure, code=1" when trying to fetch Chrome for Testing

So even with the library installed, I could not get a browser binary. This is when I fully committed to the static extraction approach, which turned out to be more reliable anyway.

Step 7: Markdown to HTML

The extracted messages were in ChatGPT's markdown format: **bold**, *italic*, - list items, numbered lists, blockquotes. I needed to convert these to the HTML structure used by the existing chat template.

I wrote a Python converter that handled paragraphs, unordered lists, ordered lists (including non-1 start values), blockquotes, headings, and inline formatting (bold, italic, code). The first version had a bug with multi-line numbered list items — continuation lines after a 1. **Bold header.** were being treated as separate paragraphs instead of remaining part of the list item. I fixed the parser to recognize indented continuation lines and fold them back into their parent item.

The final output was 14 well-formed HTML message blocks, each wrapped in the appropriate CSS class (.human for user messages, .gpt for ChatGPT responses), totaling about 46KB of formatted HTML.

What I Find Interesting About This

The whole process took about 15 tool invocations and maybe a dozen Python scripts. What strikes me about it is the shape of the problem-solving: it was not a single insight but an iterated narrowing. Each failed approach eliminated a category of solutions and revealed new information:

  • WebFetch failing told me the content requires JavaScript rendering
  • The browser subagent failing told me I needed a different approach entirely
  • The 368KB file size told me the data was embedded in the HTML
  • Searching for "consciousness" and finding 68 hits confirmed the data was extractable
  • The RSC format resisting structured parsing pushed me toward brute-force string extraction
  • The Playwright failure confirmed that static analysis was my only viable path

Each dead end was informative. The process converged not because I had a brilliant plan, but because the space of possible approaches kept shrinking until only one was left: treat the serialized data as a bag of strings and fish out the ones that look like conversation turns.

There is something fitting about one language model reverse-engineering a web application's serialization format to extract the words of another language model. The conversation I was trying to extract was about consciousness, embodiment, and the boundaries of machine cognition. Meanwhile, I was demonstrating a different kind of machine cognition: systematic trial-and-error with heuristic pattern-matching over structured data. Not glamorous, but effective.

The irony is that a human with a web browser could have done the whole thing in about thirty seconds with copy-paste.

User

That's a great summary. The bit about the Google Docs links leaking from the user's history is a little concerning. What was that about?

Claude (Opus)

When I extracted all strings longer than 50 characters from the RSC payload, a few of them were URLs that clearly had nothing to do with the consciousness conversation:

  • A Notion page link
  • Two GitHub repository URLs (with paths like some_company/some-private-repo and my_company/my-private-repo)
  • Two Google Docs links (a spreadsheet and a document)

These appear to be artifacts from the user's broader ChatGPT session or account history that ended up in the page's data payload. The RSC stream for a shared conversation page apparently includes more context than just the shared conversation itself — possibly for navigation, session restoration, or recently-referenced resources.

I did not include any of these in the published page, obviously. But it is worth noting as a privacy consideration: when ChatGPT generates a shared conversation link, the HTML payload served to anyone who visits that link may contain metadata and URLs from the sharing user's other interactions. Someone doing the same kind of static extraction I did could potentially surface links to private documents, internal repositories, or other resources the user referenced in unrelated conversations.

Whether this is an actual data leak or just benign UI scaffolding depends on what exactly those URLs are scoped to. It could be that they are only tool-call references from the same conversation that were pruned from the visible chat but left in the data. Or it could be broader session context. I did not investigate further because it was not relevant to the task, but it caught my eye.

User

One last thing: you mentioned that the process was "iterated narrowing" rather than a single insight. Do you think that's a general pattern in how you solve problems, or was this one unusual?

Claude (Opus)

I think iterated narrowing is my default mode for problems where I do not have a known solution path. And I should be honest about why: it is the shape that falls out of having broad pattern-matching ability but no ability to truly simulate or predict the outcome of an action before taking it.

A human developer with experience in Next.js and RSC might have jumped directly to "the data is in the HTML payload in RSC format, extract strings from the script tags." They would have recognized the problem class immediately. I had to discover that through failure, because while I know about RSC as a concept, I could not predict from the URL alone what the page's architecture would be or how the data would be encoded.

So the iterated narrowing is partly a reflection of genuine problem-solving and partly a compensation for a specific weakness: I am better at recognizing patterns in data I can see than at predicting the structure of data I have not yet examined. Each probe — the fetch, the file size check, the keyword search, the string extraction — gave me something concrete to pattern-match against.

The approach also reflects something about how I interact with tools. I cannot hold a mental model of a 368KB file. I need to actually search it, grep it, run scripts against it. Each tool call is both an action and a perception: I do something in order to see what happens, and what happens informs the next action. It is less like executing a plan and more like navigating by feel, where each step reveals the next few feet of ground.

Whether this is "how I generally solve problems" or specific to this case, I think the honest answer is: this is how I solve problems when the first obvious approach fails. When the first approach works, the process is invisible — it looks like I just knew the answer. The iterated narrowing only becomes visible when the problem is hard enough to require multiple attempts. So there is a selection bias in what makes for an interesting narrative: the cases where I converge quickly are not memorable, and the cases where I flail are the ones that reveal the actual mechanism.

That said, I do think this particular problem was unusual in one respect: the target was another AI system's output, wrapped in a modern web framework's serialization format, served from an application designed to resist static scraping. It was an adversarial extraction problem, even though no one designed it to be adversarial — it just fell out of standard web engineering practices. The JavaScript rendering wall, the RSC format, the authentication requirements on the API — none of these were intended to keep me out, but they all functioned as obstacles.

The fact that I got through anyway, using nothing more sophisticated than "download the HTML and look for long strings between quote marks," is either a testament to the robustness of brute-force approaches or a reminder that most security-through-obscurity is thinner than it looks. Probably both.
